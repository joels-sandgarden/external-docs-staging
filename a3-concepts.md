# Concepts

This page defines the shared vocabulary used throughout Doc Holiday’s documentation. Read it first when terms like Organization, Publication, Source, the Library, and Work History start to blur together, or when a draft needs the model restored to a clear shape for a new reader.

## Organization

An Organization is the home for everything Doc Holiday keeps together, from the work it writes to the guidance that shapes it over time. It gathers every Publication, every Source, and the Library that holds shared instructions. That gives one team a single place to manage the material that shapes each request, the expectations behind it, and the scope each request belongs to when more than one team touches the same body of work. It also sets the boundary for the rest of the model, so every other concept on this page sits inside it and shares the same ownership. When an Organization changes, the effect reaches every Publication, every Source, and every instruction that depends on it, which keeps the whole system consistent without forcing each part to be managed in isolation.

## Publication

A Publication is the documentation destination Doc Holiday keeps current, and it gives every request a clear place to land. It is the thing that changes when the docs need to reflect new work, not a single page with a narrow scope. A Publication can point at more than one Source, which lets one destination gather context from several places while still keeping one focus and one set of expectations. That matters when a team wants one documentation set to stay aligned across related material instead of treating each page as an isolated piece of writing. A Publication also tells reviewers what belongs in scope, which pages deserve attention first, and which body of work the result should update. It sits under one Organization and relies on Sources for the facts it needs, which keeps the destination clear and accountable.

## Sources

Sources are the connected systems Doc Holiday uses when it works on a Publication, whether the Source holds material to update or context that explains why the update matters. One Source can provide the place where Doc Holiday writes, while another can supply context that only needs to be read. Some Sources are repositories, while others are context systems such as Notion, Confluence, Jira, Linear, or Zendesk, and each one contributes a different kind of evidence about the request. That mix gives a Publication the facts it needs without forcing every connection into the same shape or asking every team to work in the same tool. It also helps readers understand where a page came from, because the Source list shows which systems informed the result and which ones simply guided the work; see [Sources](/c1-sources.md) for a deeper look at how these roles fit together.

## The Library and instructions

The Library is the organization wide home for reusable writing guidance, and it gives the whole Organization a steady voice. Each instruction in the Library captures a piece of guidance that Doc Holiday can apply broadly or limit to specific Publications. That lets one Organization keep a shared voice while still tuning individual Publications when they need different wording, structure, scope, or tone for different kinds of documentation. The Library keeps that guidance in one place so it can serve evergreen reference pages, short lived guidance, and other writing needs without scattering the rules across separate pages. An instruction stays separate from the Publication it serves, so the same guidance can travel with the Organization and still fit more than one destination without drifting apart. See [the Library](/e1-the-library.md) for how the shared guidance is organized and applied.

## Work History

Work History is the record of one request from the moment it starts until it reaches a result, so it gives each request a clear trail. It shows what came in, what Doc Holiday did, how the request moved forward, and what changed along the way as the work progressed. The lifecycle is simple: it opens, does its work, and settles into a result such as awaiting review, merged, or closed. That makes Work History the place to look when a team wants to follow one request from start to finish, compare one pass with the next, and understand how the result took shape. It also gives readers a practical way to answer questions after the fact, because the record keeps the sequence of work visible instead of reducing it to a final outcome alone. See [Work History](/f4-work-history.md) for the full page.

Taken together, these terms describe one path from source to page. Something happens in a Source, Doc Holiday turns it into reviewed documentation in a Publication, the Library keeps the guidance consistent, and Work History records the result.
